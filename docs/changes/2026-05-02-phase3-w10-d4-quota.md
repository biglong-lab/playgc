## 📦 新增

### 1. Endpoint：`GET /api/admin/scenarios/quota`

權限：requireAdminAuth + game:view
邏輯：
1. 取本月（從本月 1 號 0:00 起）
2. 抓所有 game.description 含 `[scenario:` 的 game
3. 場域過濾（非 super_admin 只看自己場域）
4. 配額來源優先序：
   - `SCENARIO_QUOTA_FIELD_<fieldId>` 環境變數
   - `SCENARIO_QUOTA_DEFAULT` 環境變數
   - 預設 50

回應：
```json
{
  "windowMonth": "2026-05",
  "quota": 50,
  "used": 12,
  "remaining": 38,
  "percent": 24,
  "nextResetAt": "2026-06-01T00:00:00.000Z",
  "source": "default-env"
}
```

### 2. AdminDashboard 配額卡

新增「本月情境建場配額」Card（在 Phase 2 工具卡之上）：

- 上方：`<used> / <quota>` + 百分比 + 「剩 N」
- 中間：進度條（綠 < 50% / 藍 50-80% / 琥珀 ≥ 80%）
- 下方：≥ 80% 時顯示警告「配額即將用完、下次重置時間」
- ≥ 80% 時整張卡背景轉琥珀（amber/10）

## 💡 設計決策

### 為何用環境變數而非 DB schema？

選擇：`SCENARIO_QUOTA_FIELD_<fieldId>` / `SCENARIO_QUOTA_DEFAULT`

理由：
- 不動 schema（符合紅線）
- admin 訂閱方案 = field 等級設定（不常變）
- 未來可改 DB（field_settings.scenarioQuotaPerMonth）
- 環境變數適合 MVP

### 為何不在 instantiate 端阻擋超額？

選擇：純資訊性（admin 可繼續建、不阻擋）

理由：
- W10 D4 重點是「讓 admin 知道用了多少」
- 阻擋邏輯應與付費系統整合（W10 D5+）
- 軟限制：admin 看到 80% 警告自己決定要不要升級
- 避免誤殺重要活動

### 為何顯示閾值用 80%？

選擇：≥ 80% 才轉琥珀色 + 顯示警告

理由：
- 80% 是「該開始準備升級」的甜蜜點
- 太早警告（如 50%）= 沒緊迫感
- 太晚（如 95%）= 來不及處理
- 與 disk usage 慣例一致
