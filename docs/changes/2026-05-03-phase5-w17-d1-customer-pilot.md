### 1. `docs/runbooks/customer-pilot.md`

10 步驟業務跑團 SOP（每步 ≤ 30 分鐘）：

1. **客戶確認**（10 分鐘）— 用 PitchDeck 簡報
2. **情境配對**（10 分鐘）— 引導 find-scenario 三問
3. **套餐選定**（5 分鐘）— 一次性 / 訂閱 / 委辦
4. **admin 預建場**（5 分鐘）— 電腦或 LINE
5. **活動前 1 天確認**（10 分鐘）— hostUrl + QR + 預演
6. **活動當天**（依時長）— 監控 + 突發處理
7. **活動結束**（5 分鐘）— admin 結束 + webhook 派發
8. **反饋收集**（30 分鐘）— 五個錨點問題
9. **收費**（10 分鐘）— 一次性 / 訂閱 / 委辦三流程
10. **覆盤**（30 分鐘）— 業務 + 工程閉門會議

**5 個常見 FAQ**：人數 / App / 資料保留 / 連線問題 / 客製化

**業務工具速查表**：6 個關鍵公開頁 / 後台連結

### 2. `docs/runbooks/pilot-feedback-template.md`

客戶反饋表單範本（業務複製貼上用）：

**5 個關鍵錨點**（與 ADR-0012 對齊）：
1. 整體滿意度（1-5）
2. 最受歡迎元件
3. 覺得多餘 / 不順元件

**詳細反饋區**：活動前 / 活動中 / 活動後三段

**後續客戶意願**：訂閱 / 推薦 / 案例 / 訪談
**業務改進清單**：直接驅動 customer-pilot 更新

## 💡 設計決策

### 為何 W17 主推「業務文件」而非「客戶招募」？

選擇：D1 完整 SOP、D2-D5 業務跑客戶

理由：
- 沒 SOP 直接跑 → 業務各做各的、反饋無法比較
- SOP + 反饋表 = 結構化收集 → 可規律分析
- 工程同學一天就完成 SOP、不影響業務節奏
- D2-D5 業務有完整工具支援

### 為何 10 步驟而非 5 / 20 步驟？

選擇：10 步驟、每步 ≤ 30 分鐘

理由：
- 一場活動完整流程約 5-8 小時（含活動本身）
- 10 步驟對應「一場 pilot 從頭到尾」、不會漏
- 每步 30 分鐘讓業務知道時間預估
- 太細（20 步）= 業務看不完不會用
- 太粗（5 步）= 抓不住每個關鍵節點

### 為何反饋模板含「五個錨點」？

選擇：五個必答問題對齊 ADR-0012 商業驗證

理由：
- ADR-0012 W21 必須回答 5 個問題
- 每場 pilot 都收 → W21 累積足夠樣本
- 不收這些 = W21 又要回頭問客戶（騷擾）
- 結構化問題 → 可量化分析
