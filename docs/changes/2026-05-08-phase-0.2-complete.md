## 1. 範圍

**Phase 0 完整監測體系第 2/3 階段**：建立 WS 事件 + DB 寫入完整紀錄基礎設施。

實作目標：
- 每個 WS 事件自動進 `ws_event_log`（連線/斷線/訊息/廣播/grace/auto-leave/kick）
- 多人遊戲關鍵 DB 寫入進 `db_write_log`（含樂觀鎖衝突）
- 90 天 retention（user 決定、爭議仲裁長期需求）
- 對講機訊息預設不存內容（user 決定隱私優先）
- 敏感欄位自動 redact（password / token / secret / api_key）
- Fire-and-forget + buffer + 1s flush（不阻塞 ws 處理）

## 2. 完成清單

### Schema（`shared/schema/observability.ts`、~190 行）

兩張新表：

**`ws_event_log`** — WS 事件完整紀錄
- 欄位：timestamp / sessionId / teamId / userId / userName / eventType / direction / messageType / payload / clientIp / userAgent / closeCode / reason / latencyMs / recipientCount
- eventType 列舉：connect / close / message / broadcast / error / grace_start / grace_expired / auto_leave / reconnect / kick
- direction：inbound / outbound / system
- 5 個 indexes：sessionTime / teamTime / userTime / typeTime / cleanupTime

**`db_write_log`** — DB 寫入紀錄
- 欄位：timestamp / tableName / operation / primaryKey / sessionId / teamId / userId / before / after / conflictType / retrySucceeded / triggeredBy
- 5 個 indexes：sessionTime / teamTime / tableTime / conflictTime / cleanupTime

### Migration（`migrations/manual/2026-05-08-observability.sql`、~80 行）

- 兩張表 + 10 個 indexes
- `cleanup_observability_logs(retention_days)` PL/pgSQL 函式（90 天 default）
- dev DB 已 applied

### Server Logger（`server/lib/ws-event-logger.ts`、~290 行）

- `logWsEvent()` / `logDbWrite()` 公開 API
- **Fire-and-forget**：所有 log 寫入不 await、catch all
- **Buffer**：50 筆批次 flush、或 1 秒間隔 flush
- **Payload 限制**：> 10KB 自動截斷（保留 preview）
- **隱私保護**：對講機訊息（chat / team_chat）預設不存內容、`ENABLE_CHAT_FULL_LOG=true` 才開
- **敏感欄位 redact**：password / passwd / *token / secret / api_key / credit_card 自動換成 `[REDACTED]`

### Cleanup Cron（`server/lib/observability-cleanup-cron.ts`、~95 行）

- 每天 03:00 跑、刪 90 天前資料
- 大量刪除告警（> 100 萬筆 → Telegram + 建議升 partitioning）
- `runObservabilityCleanupNow()` 手動觸發 API

### Server WS Handler 全 hook（`server/routes/websocket.ts`）

- `wss.on("connection")` → log connect + 取 client IP / user agent
- `ws.on("message")` 解析後 → log inbound message（除 keepalive）
- `ws.on("close")` → log close + closeCode + reason
- `startGraceTimer` → log grace_start
- 寬限期 expired → log grace_expired
- auto-leave timer → log auto_leave
- `kickUserFromTeam` → log kick + recipientCount
- `broadcastToSession` / `broadcastToTeam` / `broadcastToMatch` / `broadcastToBattleSlot` → log broadcast + recipientCount

### Boot 整合（`server/index.ts`）

- 加 `startObservabilityCleanupCron()` 啟動

## 4. 預估資料量（90 天 retention）

| 場景 | 估算 |
|------|------|
| 1 場活動 5 人 30 分鐘 | ~5000 events / 100 broadcasts / 50 grace 等 ≈ 1MB |
| 1 天 100 人活動 | ~100 萬 events ≈ 1GB |
| 90 天 | ~90GB |

**監控**：
- 表大小 > 50GB → Telegram 警示
- > 100 萬筆/天 cleanup 量 → 建議升 PostgreSQL declarative partitioning by day

## 5. 隱私規範

### 預設不存（對講機）
- `chat` / `team_chat` 訊息類型
- 只存 `messageLength`、不存 `message` 內容
- `_redacted: "chat_content_disabled"` 標記

### 預設 redact（敏感欄位）
- `password` / `passwd`
- `*_token` / `*Token`（含 firebaseToken / hostToken / accessToken）
- `*_secret` / `*Secret`
- `*_key` / `*Key`（API key）
- `creditCard` / `cardNumber`

### admin opt-in
- `ENABLE_CHAT_FULL_LOG=true` env var → 對講機完整內容會存（爭議仲裁需要時）
- 開啟前需業主同意條款（隱私聲明）
