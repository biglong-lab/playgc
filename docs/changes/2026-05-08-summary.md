## 3. 三大成果

### 3.1 架構治本（Phase 0-4）

**改前**：
```
玩家 Hung 進入遊戲頁
  ├─ useTeamWebSocket    new WebSocket()
  ├─ useHostScreenSync   new WebSocket()
  ├─ useTeamShootingSync new WebSocket()
  └─ ChatPanel           new WebSocket()
  → server 看到 4 條 ws、isUserStillConnected 競態誤判
```

**改後**：
```
玩家 Hung 進入任何頁
  └─ WebSocketProvider (App 層、整 app lifetime)
       ├─ useTeamWebSocket    ──┐
       ├─ useHostScreenSync   ──┼─ 全部從 Provider 拿同一條
       ├─ useTeamShootingSync ──┤
       └─ ChatPanel           ──┘
  → server 看到 1 條 ws、不會誤判
```

### 3.2 完整監測（Phase 0.2 + 0.3 + admin v2）

**3 層監測架構**：
- **Layer 1：即時觀測** — `/admin/multi-sessions`（5s refresh、3 列卡片、4 MetricCard）
- **Layer 2：完整事件 log** — `ws_event_log` + `db_write_log`（90 天 retention、async 寫入、敏感欄位 redact）
- **Layer 3：Session Replay** — `/admin/sessions/:id/replay`（時間軸 + 篩選 + CSV export）

**爭議仲裁能力**：
- 媽媽問「我兒子那場明明答對 q3、為什麼沒分數」
- → admin 看 `trivia_answers` 表 + Session Replay → 給家屬看時間戳依據

### 3.3 規範訂定（ADR-0018 + CI + 紅線）

**ADR-0018 7 條規則**：
1. 整 app 全域只能 1 條 WebSocket
2. 所有即時通訊必須透過 useWebSocket() Provider
3. 禁止直接 new WebSocket()（CI 阻擋）
4. 計分 / 排名 / 結算 → server-side source-of-truth
5. 訊息類型加新對齊 ADR-0014
6. 所有 ws 事件自動進 ws_event_log
7. 對講機等敏感訊息預設不存內容

**CI 自動化**：
- `scripts/check-ws-singleton.sh` — grep new WebSocket() 違規退出
- `e2e/global-ws-provider.spec.ts` — ADR-0018 規範驗證
- `e2e/multi-realtime-stability-phase04.spec.ts` — 真實多人 e2e

## 5. 業主新增能力

### 即時可用
1. `/admin/multi-sessions` — **三列即時連線監控**（v2 16 項打磨）
   - 4 MetricCard（進行中 / 在線 / 異常 / 平均連線）
   - 真實 online 狀態（30s/5min/offline 三態）
   - 異常 session 自動排序到頂
   - 篩選（搜尋 / 場域 / 健康度）
   - CSV 匯出
2. `/admin/sessions/:id/replay` — **完整時間軸 + 爭議仲裁**
   - 篩選 by user / event type / time range
   - 事件展開看 payload
   - CSV export
3. **Telegram 自動告警**（critical session → notify + Replay 連結 + 30min cooldown）
4. **點玩家頭像** → cross-session 歷史 dialog（看是否「常斷線玩家」）

### 隱形能力
5. **1 user = 1 條 ws**（進入遊戲不斷線、對講機自動重連）
6. **猜謎絕對公平**（server-side scoring + DB persistence）
7. **90 天事件 log**（任何時間可查爭議）
8. **CI 規範防回頭**（未來開發者寫新 ws 元件 CI 自動阻擋）
