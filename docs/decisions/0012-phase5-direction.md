# ADR-0012: Phase 5 方向 — 平台化推進與商業驗證

> **日期**：2026-05-03
> **狀態**：採用中
> **影響**：Phase 5 全週（W17-W20）走向

---

## 背景

Phase 4 完成 LINE 工具鏈（W13-W16），admin 體驗從「電腦端」搬到「手機隨身」。

接下來的核心問題：
1. 工具完整、但**真實客戶**還沒進來 — 平台價值未驗證
2. 12 情境覆蓋常見場景、但**元件庫**還能擴充（公部門 / 私部門 / 空間活化等多市場深化）
3. 平台**核心穩定性**已驗證（48 smoke test 全綠），但**觀測性**（活動健康度 / 用量趨勢）還缺
4. 商業模式設計需要**真實使用數據**才能調整定價 / 包裝

使用者願景：
> 「採用平台化的方式來做推進，提供穩定的核心與好用的元件，搭建好用的使用情境
>  公部門：街區商圈、景點串聯、空間活化
>  私部門：企業內訓、企業員工旅遊、提升團隊互動
>  活動：破冰、熱場、群體投票答題、園遊會
>  空間：遊戲腳本、各種創意發想
>  交誼：婚禮、生日、聚會」

---

## 選項

### 方案 A：全力客戶招募（4 週都跟業務）
| 優點 | 缺點 |
|------|------|
| 最快驗證商業模式 | 元件庫不擴充、多客戶撞同情境體驗單調 |
| 真實反饋驅動下個版本 | 工程同學閒置 |

### 方案 B：全力工程擴充（4 週都做元件 + 情境）
| 優點 | 缺點 |
|------|------|
| 平台技術深度增 | 沒客戶驗證、可能做白工 |
| 工程同學產出高 | 商業時間視窗壓縮 |

### 方案 C：客戶 / 工程平衡（W17 業務、W18-19 元件、W20 觀測）
| 優點 | 缺點 |
|------|------|
| 業務 + 工程都推進 | 業務 1 週時間可能不夠 |
| 元件擴充快 | 觀測延後到 W20 才做 |

### 方案 D：併行混合（每週一半業務一半工程）
| 優點 | 缺點 |
|------|------|
| 持續取得反饋 | 工程持續被打斷、效率低 |
| 每週都有客戶體驗 | 業務同學很累 |

---

## 決定

**採方案 C：客戶 / 工程平衡 + W20 觀測**

理由（≤ 5 點）：
1. Phase 4 工具完整、可承接客戶測試（先驗證再優化）
2. 工程深化需專注時間（W18-19 連續做元件 + 情境）
3. 觀測（W20）是平台長期價值核心、必須做但可後置
4. 業務 1 週密集效率比 4 週分散高
5. 每週仍有部署 + smoke test、不斷積累穩定性

### Phase 5 五週細節（W17-W20，加 W21 緩衝）

| 週 | 主題 | 範圍 |
|----|------|------|
| W17 | 真實客戶 + 第一場付費活動 | 業務找 1-3 個客戶、跑完整流程（含付費 / 反饋）|
| W18 | 元件庫擴充 5 個 host 新元件 | 投票 V2、抽獎、跑馬燈、進度條、Leaderboard 變體 |
| W19 | 情境模板擴充（12 → 20+）| 加：景點串聯、空間活化、員工旅遊、團體投票、園遊會專版 |
| W20 | 觀測 + 健康度儀表板 | admin 看：活動數 / 玩家活躍 / 收入趨勢 / 失敗 alert |
| W21 | 緩衝 + Phase 6 規劃 | 真實使用反饋反映到下一階段 |

## 影響

### 程式碼對應

**W17 業務**：
- 不寫程式（除非客戶反饋必要）
- 寫業務跑團 runbook：`docs/runbooks/customer-pilot.md`
- 收集反饋到 `docs/changes/2026-XX-XX-pilot-feedback.md`

**W18 元件擴充**：
- `client/src/components/host-screens/*.tsx` 新增 5 個
- 對應 page-types.ts / page-renderer.ts 註冊
- ShowcaseHub demo 補對應元件
- 每個元件 ≥ 5 個單元測試

**W19 情境擴充**：
- `shared/scenario-templates.ts` 新增 8+ 情境
- `getDefaultConfigForPageType` 補新 pageType 預設
- 情境模板市集 page 對應展示

**W20 觀測**：
- 新建 `client/src/pages/AdminAnalytics.tsx`
- 新增 endpoint `/api/admin/analytics/{events,revenue,health}`
- LINE 主動推播：每日 8am 健康度日報（admin 一條訊息看全貌）

### 紅線

- ✅ Schema 只新增不刪除（W17-W20 不變動既有 schema）
- ✅ Phase 4 工具不影響（向後相容）
- ✅ 業務不改 admin auth 流程（保持環境變數白名單到 W21）

### 已知限制

- W17 業務反饋可能要求改 W18-19 計畫（保持彈性）
- 元件擴充上限：每個元件 < 800 行（強制要求）
- 情境模板 > 20 後 ShowcaseHub 需分頁 / 篩選

## 後續可能變動

什麼情境會讓我們重新評估？
- **W17 業務找不到客戶** → 改方案 B 或重新檢視商業模式
- **客戶要求 schema 變動**（如多語言）→ Phase 6 才動
- **元件數突破 60** → 元件市集化（Phase 6+）
- **單場域 admin > 20** → 補 admin schema（Phase 6 開始）

---

## 商業模式驗證錨點（Phase 5 必須回答）

W21 結束時必須有答案：
1. **付費客戶數**：W17 + W20 內成交多少場活動？
2. **平均單價**：一次性 / 訂閱 / 委辦的實際成交價？
3. **回購率**：第一次付費客戶有多少回流？
4. **NPS**：客戶推薦給其他客戶的可能性？
5. **元件熱度**：12 + 8 = 20 情境中、哪 3-5 個被使用 80%？

這些數據會直接決定 Phase 6 方向（深化 / 擴充 / pivot）。
