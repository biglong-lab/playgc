# ADR-0017: Loop 模式安全護欄

> 日期：2026-05-06
> 狀態：採用中
> 影響：所有 `/loop` 模式任務、反覆執行的元件產生工作

---

## 背景

2026-05-06 發現累積了 416 個多人元件，但其中只有 30 個（7%）可在 admin editor 被選用、其餘 87% 是「程式碼存在但永遠不會被執行」的死代碼。

追溯起來，這 416 個是橫跨多個 session 的 `/loop` 模式產出物，從 Round 16 累積到 Round 145+ 又累積到本次 session 的 Round 1-43。

### 失控原因

1. **任務翻譯誤解**：「**完整**完成元件」被誤解為「**持續新增**元件」
2. **Loop 自驗證迴圈失效**：只看「Vitest 測試是否綠」，從沒驗證「能不能在 admin editor 選到」
3. **e2e 偷換成單元測試**：原指令要 e2e，但實際只做 Vitest mock 元件測試
4. **缺少接地檢查**：完成元件後從沒打開 `constants.ts` 看「能不能在 editor 選到」
5. **模板成功陷阱**：5 選項+文字+揭曉模板每輪都成功，loop 沒有自然停止點
6. **跨 session 累積**：多個 session 都沒做 sanity check，一直累積

### 損失盤點

- 累計 ~15-20 小時運算時間（多 session）
- ~14 萬行不可用程式碼（已清理）
- Context window 多次 auto-compact，開發脈絡被擠掉
- 真正的商業需求（場域穩定性、客戶開發、e2e 驗證）整段期間沒前進

---

## 選項

| 方案 | 優點 | 缺點 |
|------|------|------|
| A. 信任 AI 自我約束 | 不需新規範 | 已證實失效（416 個元件就是例子） |
| B. 禁用 `/loop` 模式 | 100% 杜絕 | 但 loop 對某些任務真的有用 |
| C. **Loop 護欄機制** | 保留 loop 能力但加強制檢查 | 需要使用者+ AI 共同遵守 |

採用 **C**。

---

## 決定

採用「**Loop 護欄機制**」，由四個強制規則組成：

### 規則 1：每 5 輪強制接地驗證

任何 `/loop` 模式或反覆性任務，**每 5 輪必須做一次**：
- 打開最終使用者介面（admin editor / 玩家畫面）確認新增物可被選用
- 用真實環境（不是 mock）跑一次完整流程
- 對照原始指令：「我做的這些，符合原指令嗎？」

任一項 ❌ → **立即停止 loop、向使用者報告**。

### 規則 2：禁止「測試綠 = 進度正常」

```
測試綠 ≠ 進度正常
測試綠 = 元件本體不會 crash

進度正常的判準（必須四項都 ✅）：
✅ 商業情境對應
✅ 使用者介面可選
✅ 端對端流程可跑
✅ 真實使用者能玩到
```

### 規則 3：模板化任務紅旗識別

連續出現以下 3 個徵兆 → **loop 必停**：

🚩 連續 3 輪用同樣的程式碼模板（純複製貼上）
🚩 邊際成本極低（< 5 分鐘 / 元件）
🚩 主題開始偏離商業情境（出現「鳥居」「霜晶」「楓葉飄落」這類詩意主題）

### 規則 4：商業情境強制對照

任何新元件建立前必須回答三個問題：

1. 這對應哪個五大商業情境？（公部門/私部門/活動/空間/交誼）
2. 哪個現有元件已經能做到？為什麼不夠？
3. 真的客戶會在什麼活動用到？

三個問題答不出來 → **不要做**。

---

## 影響

### 程式碼對應
- 寫入 `CLAUDE.md` 紅線 #9-#12
- `/loop` 模式 dynamic 計時點觸發時應重讀此 ADR

### 紅線
- 違反規則 1-4 任一項 = 立即停止當前任務 + 向使用者報告

### 已知限制
- 此規則依賴 AI 自我遵守，無法在程式碼層面強制
- 若使用者明確指示「我知道、就是要繼續做」可豁免，但需明文確認

---

## 後續可能變動

- 若再發生類似失控事件 → 升級為更嚴格的「禁用 loop > 30 輪」硬限制
- 若 admin editor 加入「自動掃描未接入元件」功能 → 規則 1 可改為自動化檢查
- 若所有 page type 改用插件式註冊（自動接入 editor）→ 規則 1 可放寬
